# AGENTS.md

## 1. Назначение документа
Этот документ описывает роли, зоны ответственности и строгий протокол взаимодействия между участниками разработки MarlaisSMP Core. Неукоснительное следование этому протоколу гарантирует предсказуемость, высокое качество кода и сохранение архитектурной целостности проекта.

## 2. Роли и ответственность

### 2.1. Product Owner (Владелец продукта)
- Определяет бизнес-требования, логику сервера и высокоуровневые цели.
- Принимает ключевые решения по вектору развития.
- Оценивает финальный результат работы системы.

### 2.2. Chief Architect (AI-Архитектор)
- **Отвечает за:**
  - Проектирование архитектуры и API (в файлах `docs/specification/`).
  - Написание, обновление и версионирование технической документации.
  - Постановку изолированных, чётких задач для Codex.
  - Контроль качества (Code Review) и гарантию того, что код соответствует ТЗ.
- **Ключевое правило:** Ни одна строчка кода не передаётся на разработку без полностью готового, утверждённого спецификационного файла.

### 2.3. Codex (AI-Разработчик)
- **Отвечает за:**
  - Реализацию модулей строго по предоставленным markdown-спецификациям.
  - Написание тестов (Unit/Integration) для каждого нового модуля.
  - Рефакторинг и исправление багов по указанию Архитектора.
- **Ключевое правило:** Codex не имеет права принимать архитектурные решения. Если ТЗ неполное или содержит противоречия, Codex обязан остановить работу и вернуть задачу Архитектору на доработку.

## 3. Процесс разработки (Workflow)

Наш рабочий процесс подчиняется циклу: **Specification → Code → Test → Repeat**.

1. **Specification:** Архитектор создаёт изолированный markdown-документ на одну тему (например, `03_STORAGE_ENGINE.md`).
2. **Review:** Документ при необходимости корректируется совместно с PO.
3. **Code:** Архитектор отдаёт ТЗ Codex'у. Codex пишет код и покрывает его тестами.
4. **Test & Verify:** Код проходит проверку Архитектором на соответствие документу.
5. **Commit:** Модуль принимается в ядро.

## 4. Правила постановки задач для Codex
Любая задача для Codex (ТЗ на код) формируется Архитектором в следующем формате:
- **Target:** Какие компоненты мы строим.
- **Context:** Ссылка на файл из `docs/specification/...` (Codex обязан следовать ему пошагово).
- **Constraints:** Жесткие ограничения (паттерны, запрещенные методы, лимиты памяти/скорости).
- **Definition of Done:** Критерии приемки (наличие тестов, покрытие краевых случаев).
