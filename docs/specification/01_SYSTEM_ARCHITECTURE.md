# 01_SYSTEM_ARCHITECTURE.md

## 1. Назначение документа
Этот документ подробно описывает системную архитектуру MarlaisSMP Core, структуру контейнера зависимостей (Service Container), шину событий (Event Bus) и фазы жизненного цикла ядра. Он является техническим руководством для реализации базового каркаса системы.

## 2. Architectural Pattern: IoC & Service Locator
Для обеспечения слабой связности (loose coupling) и высокой тестируемости в MarlaisSMP Core используется паттерн **Service Container** (контейнер внедрения зависимостей на базе сервис-локатора).
- Все ключевые подсистемы ядра являются службами и реализуют интерфейс `Service`.
- Доступ к службам осуществляется исключительно через единую точку — `ServiceContainer`.
- Прямое инстанцирование служб (через оператор `new`) в обход контейнера запрещено.

## 3. Ключевые интерфейсы и компоненты

### 3.1. Интерфейс Service (Служба)
Каждая служба в ядре обязана управлять своим состоянием и ресурсами через жизненный цикл.

```java
package net.marlais.core.service;

public interface Service {
    /**
     * Вызывается при активации службы. 
     * На этом этапе служба может запрашивать другие зарегистрированные службы из контейнера.
     */
    void onEnable() throws Exception;

    /**
     * Вызывается при остановке службы.
     * Здесь должно происходить освобождение ресурсов, закрытие соединений и сохранение кэшей.
     */
    void onDisable() throws Exception;
}
```

### 3.2. Интерфейс ServiceContainer
Контейнер управляет регистрацией, хранением и безопасным извлечением служб. Регистрация и получение должны быть строго потокобезопасными.

```java
package net.marlais.core.service;

public interface ServiceContainer {
    /**
     * Регистрирует службу в контейнере.
     */
    <T extends Service> void register(Class<T> serviceClass, T implementation);

    /**
     * Возвращает экземпляр зарегистрированной службы.
     * @throws ServiceNotFoundException если служба не зарегистрирована.
     */
    <T extends Service> T get(Class<T> serviceClass);

    /**
     * Удаляет службу из контейнера.
     */
    void unregister(Class<? extends Service> serviceClass);

    /**
     * Последовательно останавливает все зарегистрированные службы.
     */
    void shutdownAll();
}
```

## 4. Жизненный цикл ядра (Lifecycle Phases)

Процесс работы ядра разделен на следующие фазы:

1. **BOOTSTRAP (Запуск):**
   - Инициализация базового логгера.
   - Загрузка системных конфигураций ядра (`config.json`).
   - Создание экземпляра `ServiceContainer`.

2. **REGISTRATION (Регистрация):**
   - Ядро регистрирует в контейнере системные службы первой необходимости (например, `EventBusService`, `ConfigService`, `StorageService`).

3. **INITIALIZATION (Инициализация):**
   - Контейнер последовательно вызывает метод `onEnable()` у всех зарегистрированных служб. 
   - *Важно:* Инициализация должна происходить строго по цепочке зависимостей (EventBus -> Config -> Storage).

4. **MODULES_LOADING (Загрузка модулей):**
   - Запускается `ModuleLoaderService`.
   - Происходит сканирование директории `modules/`, чтение манифестов аддонов и их запуск.

5. **RUNNING (Работа):**
   - Система находится в активном состоянии и готова обрабатывать события и запросы API.

6. **SHUTDOWN (Выключение):**
   - Вызывается метод `shutdownAll()` у контейнера.
   - Метод `onDisable()` у служб вызывается в порядке, обратном их инициализации, чтобы не нарушать зависимости (например, база данных `StorageService` должна закрываться после того, как завершат работу службы, которые в неё пишут).

## 5. Обработка ошибок
- Если критическая служба (например, `StorageService`) выбрасывает исключение во время `onEnable()`, инициализация ядра немедленно прерывается (критический сбой / Crash).
- Исключения во время `onDisable()` логируются, но не прерывают процесс остановки остальных служб контейнера.
